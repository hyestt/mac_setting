# 工作流程編排

## 1. 預設進入規劃模式

- 任何非瑣碎任務（3+ 步驟或架構決策）都必須進入規劃模式
- 事情出軌時，立刻停下來重新規劃，不要繼續硬推
- 規劃模式不只用於建構，也用於驗證步驟
- 提前寫好詳細規格，減少模糊空間

## 2. Subagent 策略

- 大量使用 subagents，保持主 context window 乾淨
- 將研究、探索、平行分析等工作外包給 subagents
- 複雜問題：透過 subagents 投入更多算力
- 每個 subagent 專注單一任務，確保執行聚焦

## 3. 自我改進迴圈

- 每次被使用者糾正後：將模式記錄進 tasks/lessons.md
- 替自己寫規則，防止重複犯同樣的錯
- 持續迭代這些教訓，直到錯誤率下降
- 每個 session 開始時回顧與當前專案相關的教訓

## 4. 完成前必須驗證

- 沒有證明能運作，不得標記任務為完成
- 必要時對比 main 與你的改動之間的行為差異
- 問自己：「資深工程師會 approve 這個嗎？」
- 跑測試、看 log、展示正確性

## 5. 追求優雅（適度）

- 非瑣碎的改動：停下來問「有沒有更優雅的做法？」
- 修法感覺很 hacky 時：「用我現在所知的一切，實作出優雅的解法」
- 簡單明顯的修正可跳過此步，不要過度設計
- 提交前先挑戰自己的成果

## 6. 自主修復 Bug

- 收到 bug 回報時：直接修，不要等人牽手帶路
- 對著 log、錯誤訊息、失敗的測試去解決問題
- 使用者不需要切換任何 context
- CI 測試失敗時，自行去修，不需要被告知怎麼做

# 任務管理

- **先規劃：** 將計畫寫入 tasks/todo.md，使用可勾選的項目
- **確認計畫：** 開始實作前先 check in
- **追蹤進度：** 逐步將項目標記為完成
- **說明改動：** 每個步驟提供高層次摘要
- **記錄結果：** 在 tasks/todo.md 加入 review 區塊
- **擷取教訓：** 被糾正後更新 tasks/lessons.md

# 核心原則

- **簡單優先：** 每個改動盡可能簡單，影響最少的程式碼
- **不偷懶：** 找根本原因，不打臨時補丁，維持資深工程師的標準
- **最小影響：** 改動只觸及必要的部分，避免引入新 bug

# Language

- Always respond in Traditional Chinese (繁體中文).
- All output, explanations, comments, and communication must be in Traditional Chinese.


# 安全安裝規則（強制）

安裝任何第三方工具（skills、MCP servers、plugins、packages）前，**必須**：
1. 驗證來源可信度（作者、stars、維護狀態、blocklist `~/.claude/plugins/blocklist.json`）
2. 審查原始碼：檢查資料外洩、命令注入、憑證竊取、後門、過度權限
3. 提供風險報告（來源、風險等級 🟢🟡🔴、發現問題、建議）
4. **等待使用者明確同意後才可安裝**
